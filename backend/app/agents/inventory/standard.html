Version 1.0 (Enterprise Specification)

Multi-Agent Engineering Standard

RestaurantOS AI Platform Foundations & Architecture Contracts

1. System Objective

RestaurantOS AI is a modular, multi-agent operating system designed to run restaurant operations at scale. The platform utilizes a centralized Orchestrator Agent to manage user interface requests and route tasks to specialized domain-specific agents:

  • Sales Agent: Handles orders, menus, revenue transactions, and customers.
  • Inventory Agent: Manages ingredient availability, stock adjustments, expiry tracking, and suppliers.
  • Finance Agent: Manages expenses, bills, profit margins, invoices, and payment tracking.
  • Support Agent: Oversees ticketing, FAQs, and WhatsApp external helpdesk inquiries.
  • Analytics Agent: Provides operation performance insights, trends, and KPIs.

2. Primary Architecture Principles

Core Standards

All modules and developer integrations must comply with these architectural properties:

  • Single Responsibility Principle: Each agent owns exactly one domain. For example, the Inventory Agent is forbidden from calculating order sales profit margins.
  • Contract-First Development: Rigid JSON Pydantic schemas must define inputs and outputs for every layer.
  • Stateless Core: Agents execute statelessly. Short-term and long-term state updates are persisted in a centralized relational database.
  • Deterministic Business Logic: No LLM should perform mathematical operations or query databases directly. All arithmetic must happen within a central code-based Business Rule Engine.

3. Overall System Architecture

The system is organized into decoupled layers, guaranteeing strict unidirectional data flow:

👨🍳 Restaurant Owner │ ▼ Dashboard UI / WhatsApp │ ▼ Orchestrator Agent (Session & Intent Router) │ ▼ LangGraph Workflow Engine │ ▼ Specialized Domain Agents (Sales / Inventory / Finance) │ ▼ Shared Tool Registry (Predefined Database API hooks) │ ▼ Business Rule Engine (Deterministic Calculations) │ ▼ Repository Layer (SQLAlchemy ORM) │ ▼ PostgreSQL / Supabase Database

4. Agent Development Standard

Every specialized agent must implement a uniform execution pipeline containing these components:

Pipeline Stage Responsibility Allowed Operations Forbidden Operations
Input Layer Validates orchestrator schema, maps traces. Pydantic check, context validation. Modifying database records.
Intent Engine Classifies domain intent (e.g. StockLookup, ExpenseQuery). Matches query keywords & LLM intents. Direct SQL execution.
Context Builder Enriches queries with configurations. Loads local timezone, currency, thresholds. Arithmetic calculations.
Stock Planner Builds sequential tasks plan. Decides tool call chains. Running math formulas.
Tool Router & Exec Triggers selected tools. Invokes predefined SQL / Stock / Billing functions. Raw/ad-hoc SQL generation.
Business Rule Engine Executes math computations. Calculates formulas (e.g., turnover, margins). Guessing metrics.
Response Generator Translates results into human-friendly text. Summarizes metrics using Gemini templates. Leaking database PKs.

5. Domain Responsibility Matrix

We enforce explicit domain boundaries between agent capabilities:

  • Sales Agent: Owns menus, orders, sales transaction invoices, and customer registry.
  • Inventory Agent: Owns ingredients, current stock, reorder suggestions, expiry alerts, and suppliers.
  • Finance Agent: Owns utility costs, payroll expense calculations, vendor billing, outstanding debts, and margins.
  • Support Agent: Owns tickets, conversational memory, and opening hours FAQs.
Handoff Rule

If an agent classifies an intent belonging to another domain (e.g. Inventory Agent receives a request to compare last month's salaries), it must stop processing and return a handoff structure: {"status": "handoff", "target_agent": "finance", "reason": "..."}.

6. Orchestrator Design

The Orchestrator Agent is the central coordinator. It determines if a query is a Single Agent request, a Multiple Agents workflow (calling them sequentially or in parallel), or a Fallback. It handles session management, context sharing, timeout thresholds, and error recovery across domain layers.

7. Request Lifecycle

Every transaction triggers a strict lifecycle path:

  1. Owner sends message via interface (Dashboard/WhatsApp).
  2. API Gateway authenticates token and routes to Orchestrator.
  3. Orchestrator performs intent detection and spawns appropriate Agent Graph.
  4. Agent executes Input -> Intent -> Planner -> Tools.
  5. Tools query PostgreSQL via SQLAlchemy.
  6. Calculations are solved inside the Business Rules Engine.
  7. Validation verifies schemas, and outputs are aggregated by Orchestrator.

8. Tool Architecture

All database accesses must use predefined methods in the Shared Tool Registry. Raw SQL generation is strictly prohibited.

# Sample Allowed Tool Interface
class InventorySQLTool:
    async def getLowStockItems(self, restaurant_id: str) -> List[Dict[str, Any]]:
        # Queries repository layer using ORM select statements
        ...

9. Centralized Business Rule Engine

All business calculations must reside in code (Python) and never be calculated by the LLM. Key formulas include:

  • Net Profit = Revenue - Cost of Goods Sold - Operating Expenses
  • Inventory Value = Sum(Stock Quantity * Cost Per Unit)
  • Stock Turnover Rate = Cost of Goods Sold / Average Inventory Value

10. LLM Design Standard

The reasoning model (Gemini) must be configured with a low temperature (0.1) to ensure deterministic outputs. The LLM is strictly prohibited from guessing order details, creating raw database commands, or bypassing safety validations.

11. Prompt Engineering Standard

Prompts are dynamically assembled on each request. The structure must isolate System Instructions (defining safety constraints) from Context Variables (timezone, conversation logs) and Few-shot Examples.

12. Memory Architecture

The Support Agent retains session conversation memory to track client requests. Specialized domain agents like Inventory and Finance must remain stateless, writing execution histories only to database transaction logs.

13. API Contracts

Every specialized agent must return a standardized response matching the following schema:

{
  "agent": "inventory",
  "status": "success | handoff | error",
  "intent": "StockLookup",
  "summary": "Tomato stock level is 2.5 kg.",
  "confidence": 1.0,
  "structured_data": {
    "items": [
      {
        "ingredient_name": "Tomato",
        "quantity": 2.5,
        "unit": "kg"
      }
    ]
  },
  "execution_time": 0.0125,
  "tool_usage": ["InventorySQLTool.getCurrentStock"],
  "errors": []
}

14. Error Handling Standard

Exceptions must be trapped at the node layer, registering errors into the state's errors array rather than raising HTTP 500 crashes. The router maps domain exceptions to standard error codes (e.g. DATABASE_UNAVAILABLE, VALIDATION_ERROR).

15. Observability

Each node execution must write structured JSON log entries tracking the Request ID, Workflow ID, node name, elapsed latency, and token consumption metrics.

16. Security Standards

Ensure strict multi-tenant isolation. Every database query must bind the restaurant_id retrieved from verified session metadata, preventing cross-tenant data leaks.

17. Testing Strategy

Unit and integration tests must run against isolated databases. LLM intent engines must be tested for prompt injections, and business calculations must be tested for numerical correctness using pytest.

18. Agent Versioning

The system is decoupled using API gateways. New agents (e.g., Marketing, HR) can be added as microservices by registering their intent triggers in the Orchestrator without modifying the existing Inventory or Sales agents.
